# Platform Event Log

`public.platform_events` is the append-only event spine for Forget About. Any significant user or system action should insert a new row rather than editing an existing one.

## Row Contract

Each event row includes:

- `id`: UUID primary key, generated by Postgres.
- `occurred_at`: UTC `timestamptz`, generated when the row is inserted.
- `event_type`: dot-namespaced event name, for example `factory.order.completed`.
- `entity_type`: the business object type the event is about.
- `entity_id`: the stable ID or reference for that entity.
- `user_id`: nullable user ID for the user the event belongs to.
- `actor_user_id`: nullable user ID for the user who performed the action.
- `payload`: JSONB context. Keep it useful, bounded, and free of raw secrets.
- `version`: positive integer event schema version. Start at `1`; increment only when consumers need to distinguish payload shape.

Rows must not be updated or deleted. The schema grants only `select` and `insert` to authenticated users, and a database trigger raises `platform_events is append-only` for any `update` or `delete`, including privileged service code.

## Canonical Events

These event names are the platform-level contract for dashboards, automations, finance reconciliation, and future integrations:

- `user.registered`
- `generator.started`
- `generator.completed`
- `stl.exported`
- `unlock.purchased`
- `factory.order.created`
- `factory.order.accepted`
- `factory.order.completed`
- `factory.order.failed`
- `printer.onboarded`
- `printer.certified`
- `customer.review.submitted`
- `printer.review.submitted`
- `payment.completed`
- `refund.processed`

Lower-level operational events are also allowed, such as `email.queued`, `marketplace.quotes_created`, `printer.message.sent`, and `factory.order.delivery_chaser_sent`. They should not replace the canonical events above when a canonical business milestone has happened.

`printer.review.submitted` is reserved for the future provider-to-customer review flow. Do not reuse it for order notes or customer support messages.

## Publishing Events

Server code should call `recordPlatformEvent(...)`. Browser code should call `window.accountService.recordEvent(...)`, which posts to `/api/events` with the signed-in user's token. Prefer the canonical event name, set `entity_type` and `entity_id`, and put extra detail in `payload`.

Example:

```js
await recordPlatformEvent({
  eventType: "factory.order.completed",
  actorUserId: customerUserId,
  actorRole: "customer",
  brandKey: "tray",
  generatorType: "movement_tray",
  entityType: "print_job",
  entityId: printJobId,
  payload: { orderId, rating: 5 },
  version: 1
});
```

## Subscribing

Future services can subscribe to new events in three safe ways.

1. Realtime insert subscription for live services:

```js
const channel = supabase
  .channel("platform-events")
  .on(
    "postgres_changes",
    { event: "INSERT", schema: "public", table: "platform_events" },
    ({ new: event }) => {
      // Handle event.id idempotently.
    }
  )
  .subscribe();
```

The schema adds `public.platform_events` to the `supabase_realtime` publication when that publication exists. Supabase documents this pattern in its Realtime Postgres Changes guide: https://supabase.com/docs/guides/realtime/postgres-changes

2. Cursor polling for batch workers:

```text
/rest/v1/platform_events?occurred_at=gt.<last_seen_iso>&order=occurred_at.asc,id.asc
```

Store the last processed `occurred_at` and `id`, and treat `event.id` as the idempotency key.

3. Future dispatcher service:

A worker can subscribe to inserts, validate the `event_type`, and fan out to email, finance, analytics, AI decisioning, fulfilment, or CRM queues. Consumers must never mutate `platform_events`; write derived state to their own tables.
